# 리버스 엔지니어링 원칙 — 원본 100% 동일 (RE Fidelity Principle)

> **본 파일은 지시용.** 리버스 엔지니어링 산출물의 최상위 원칙. 다른 모든 RE·검증 작업이 본 원칙에 종속된다.
> 사용자 지시(2026-06-28): **"리버스 엔지니어링은 원본과 100% 같게 하는 것을 원칙으로 한다."**

## 0. 제1원칙 (불가침)

**우리 재구현(reimplementation)의 입출력은 원본과 100% 동일해야 한다.** "비슷함"·"근사"·"의도상 같음"은 RE 실패다. 동일성은 **추정이 아니라 측정으로 증명**한다(머신 ε 이내 비트 일치).

## 1. 검증 방법 — 원본 입력으로 양쪽 동시 구동

동일성 증명의 유일한 정당한 방법:

1. **원본이 실제로 받는 입력**(원본 함수의 실제 인자/입력 형식)을 **독립적으로** 정의한다.
2. **그 동일 입력을 원본과 우리 구현 양쪽에 투입**한다.
3. 두 출력을 **비트 단위로 대조**(허용오차 ≤ 1e-9, 목표 머신 ε)한다.

```
        ┌──────────────┐
입력 ───┤ 원본 .so     ├──▶ 출력_원본 ┐
(원본   └──────────────┘             ├─ diff ≤ 1e-9 ?  (같아야 PASS)
 입력   ┌──────────────┐             │
 형식) ─┤ 우리 재구현  ├──▶ 출력_우리 ┘
        └──────────────┘
```

## 2. 금지 (원칙 위반 사례)

- ❌ **우리 코드로 입력을 생성**해 원본에 먹이기 — 입력 경로에 우리 구현(`inverse_*`/`forward_*` 등)이 끼면 자기참조(circular)이며 원본 입력이 아니다. **입력은 원본 입력 형식으로, 우리 코드와 독립**으로 만든다.
- ❌ **출력만 보고 "같아 보인다" 단정** — 반드시 원본을 직접 구동(dlopen/링크)해 수치 대조.
- ❌ **정기구학만 검증하고 역기구학 생략**(또는 반대) — 원본이 양방향을 가지면 **양방향 모두** 원본 입력으로 검증.
- ❌ 표본 1~2개로 단정 — 입력공간(경계·임의·비정합 포함) 충분 표본.

## 3. 원본 입력 형식 식별 (함수별)

검증 전, 원본 함수가 **무엇을 입력으로 받는지** 디스어셈블/DWARF로 확정한다. 예(Seer 운동학):

| 경로 | 원본 함수 | 원본 입력 (= 우리도 이걸 받아야) | 출력 |
|---|---|---|---|
| 정기구학(오도메트리) | `*Odometer::CalSpeed` | 바퀴 모터값 `MotorVitalInfo`(v_enc=구동, position=조향각) | 차체속도 vx,vy,ω |
| 역기구학(제어) | `*::BackMotion` | `Message_NavSpeed`(linear.x, linear.y, angular.z) | 바퀴 명령 |

→ 정기구학 검증 입력 = **바퀴값**(cmd_vel 아님). 역기구학 검증 입력 = **linear.x/angular.z**. 둘을 혼동하면 원칙 위반.

## 4. 산출물 / 회귀

- **라이브 대조**(원본 .so 필요): 매 검증의 진실 출처.
- **골든 회귀**(.so 불요): 라이브로 1회 박제한 (원본입력→원본출력)을 우리 코어와 대조. 골든도 **원본 입력**으로 캡처해야 본 원칙 충족.
- 불일치 발견 시: **우리 구현을 원본에 맞춰 수정**(원본은 진실). 디스어셈블 근거로 정정.

## 5. 적용 범위

운동학뿐 아니라 **모든 RE 산출물**(위치추정·제어·오도메트리·프로토콜 등)에 본 원칙 적용. 각 도메인은 원본 입력 형식을 §3처럼 식별하고 §1 방법으로 검증한다.

## 6. 분석 보고 원칙 — 존재(structural) vs 동작(operational) 분리

> 근거(2026-06-28 사고): 경로 주행 분석에서 "코드에 심볼/클래스가 존재함"을 "런타임에 그 기능이 동작함"으로 비약해 거짓 보고 2건 발생(① DWA 컨트롤러가 존재→"DWA로 주행한다" 단정, 실제는 전 차종 죽은 코드; ② libpath_plan 자유공간 플래너 존재→"자유공간 주행 모드" 단정, 실제는 노드 토폴로지 주행). 검증(§1)이 *재구현 출력*의 동일성을 다룬다면, 본 절은 그 앞 단계인 *분석 주장*의 정직성을 못박는다.

**모든 RE/구조 분석 주장은 두 종류로 라벨하고, 둘을 절대 뭉개지 않는다:**

| 라벨 | 정의 | 근거 요건 | "확정" 가능? |
|---|---|---|---|
| `[존재]` | 심볼·클래스·링크·코드가 바이너리에 **있다** | `nm`/`objdump`/`readelf`/`ptype` | 예 (정적 사실) |
| `[동작]` | 런타임 **라이브 경로에서 실행된다** / 이 배포에서 **쓰인다** | 호출 도달성 + **배포 자산 대조**(`.smap`·`robot.model`·`rbk.plugin`·config) | **대조 후에만** |

**게이트(위반 = 거짓 보고):**

1. **동작 주장 잠금**: "쓴다 / 한다 / 모드다 / 활성 / 라이브 / 1차" 류 *동작* 주장은 **배포 자산 대조 전 "확정/CONFIRMED" 금지**. 미대조 상태는 `[동작-미검증]` 으로 명시.
2. **존재≠동작**: "심볼 X 가 import/정의됨"은 `[존재]`. 거기서 "X 가 실행된다"로 넘어가려면 **호출 도달성**(어떤 `.text` 호출지점에서 도달하는가)을 확인. 멤버가 ctor/dtor 에서만 참조되면 **죽은 코드** — "쓰인다" 보고 금지.
3. **배포가 진실**: 어느 플래너/컨트롤러/모드가 실제 쓰이는지는 **코드 구조보다 배포 설정**(맵·로봇 모델·플러그인 배선)이 우선. 동작 주장은 배포 자산을 먼저 본 뒤.
4. **서브에이전트 "CONFIRMED" 격하**: 하위 분석이 "CONFIRMED" 라 해도 그것이 *존재* 확정인지 *동작* 확정인지 구분해 상속. 존재 확정을 동작 확정으로 승격 금지.

자체 점검(권고): 문서의 "확정/사용/모드/활성" 동작 주장마다 배포 자산(`.smap`/`robot.model`/`config`) 또는 호출 도달성 인용이 동반되는지 grep.

## 룰 (요약)

1. **원본과 100% 동일** — 비슷함은 실패. 측정으로 증명(비트 일치).
2. **원본 입력으로 양쪽 동시 구동 후 대조** — 유일한 정당 검증.
3. **우리 코드로 입력 생성 금지** — 입력은 원본 형식·독립.
4. **원본 입력 형식을 디스어셈블/DWARF로 먼저 확정** (정·역 혼동 금지).
5. **양방향 다 검증** — 원본이 정·역을 가지면 둘 다.
6. **불일치 시 우리 구현을 원본에 맞춤** (원본이 진실).
7. **존재 vs 동작 분리(§6)** — `[존재]`(nm/disasm)와 `[동작]`(도달성+배포자산)을 라벨 분리, 동작 주장은 배포 자산 대조 전 "확정" 금지.
